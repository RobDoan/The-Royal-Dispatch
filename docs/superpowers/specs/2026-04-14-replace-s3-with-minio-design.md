# Replace AWS S3 with Self-Hosted MinIO

**Date:** 2026-04-14
**Status:** Approved
**Approach:** Direct boto3 endpoint swap (Approach A)

## Summary

Replace AWS S3 object storage with a self-hosted MinIO instance running as a Docker Compose service. MinIO is S3-compatible, so boto3 continues to work — the change is confined to configuration and URL construction.

## Context

The Royal Dispatch uses AWS S3 to store MP3 audio files generated by ElevenLabs. The backend uploads via boto3 `put_object()`, constructs a public URL, stores it in PostgreSQL, and the frontend plays it via an `<audio>` tag. The bucket is publicly readable (no presigned URLs).

The goal is to self-host storage via MinIO on the same machine as the Docker Compose stack, exposed publicly via a reverse proxy with HTTPS.

## Decisions

- **Fresh start** — no migration of existing S3 files
- **Public bucket** — same as current S3 setup, anonymous read access
- **Keep boto3** — no new Python dependencies
- **MinIO in Docker Compose** — same machine as the rest of the stack

## Design

### 1. Docker Compose — MinIO Service

Add two services to `docker-compose.yml`:

**`minio` service:**
- Image: `minio/minio`
- Command: `server /data --console-address ":9001"`
- Ports: `9000` (S3 API), `9001` (web console)
- Volume: `minio_data:/data`
- Env vars: `MINIO_ROOT_USER`, `MINIO_ROOT_PASSWORD` (from `.env`)
- Healthcheck on `/minio/health/live`

**`minio-init` service (one-shot):**
- Image: `minio/mc`
- Depends on: `minio` (healthy)
- Creates the `royal-audio` bucket if it doesn't exist
- Sets bucket policy to public (anonymous download)
- Sets CORS policy to allow `GET` from frontend origins
- `restart: "no"` (runs once)

Backend gains `depends_on: minio`.

### 2. Backend Code Changes

**`backend/storage/client.py`:**
- Add `endpoint_url=os.environ["S3_ENDPOINT_URL"]` to `boto3.client("s3", ...)`
- Read credentials from `S3_ACCESS_KEY` and `S3_SECRET_KEY` (renamed from `AWS_*`)
- Remove `region_name` parameter

**`backend/nodes/synthesize_voice.py`:**
- Replace URL construction from `f"https://{bucket}.s3.{region}.amazonaws.com/{filename}"` to `f"{os.environ['S3_PUBLIC_URL']}/{bucket}/{filename}"`
- Remove `region` variable and `AWS_REGION` reference

### 3. Environment Variables

**Old (removed):**
```
AWS_ACCESS_KEY_ID=your-key-id
AWS_SECRET_ACCESS_KEY=your-secret-key
AWS_REGION=us-east-1
S3_BUCKET=royal-audio
```

**New:**
```
S3_ACCESS_KEY=minioadmin
S3_SECRET_KEY=minioadmin
S3_ENDPOINT_URL=http://minio:9000
S3_PUBLIC_URL=https://minio.quybits.com
S3_BUCKET=royal-audio
MINIO_ROOT_USER=minioadmin
MINIO_ROOT_PASSWORD=minioadmin
```

- `S3_ENDPOINT_URL`: Used by boto3 inside Docker network (`http://minio:9000`)
- `S3_PUBLIC_URL`: Used for constructing URLs returned to the frontend (the public reverse-proxy address)
- `S3_ACCESS_KEY` / `S3_SECRET_KEY`: Same values as `MINIO_ROOT_USER` / `MINIO_ROOT_PASSWORD`
- `S3_BUCKET`: Unchanged, still `royal-audio`

### 4. Reverse Proxy & CORS

- MinIO S3 API exposed at a public domain (e.g., `minio.quybits.com`) via reverse proxy forwarding to `minio:9000`
- Optionally expose MinIO console (`minio:9001`) for admin access
- CORS configured on the `royal-audio` bucket via `minio-init` to allow `GET` from frontend origins
- Reverse proxy configuration (Nginx/Caddy) is outside scope — only the MinIO side and env vars are covered here

### 5. What Doesn't Change

- **boto3** remains the S3 client library
- **Frontend** — zero changes; receives `audio_url` from API
- **Admin UI** — no changes
- **Database** — no schema changes; `audio_url` stores whatever URL the backend produces
- **Upload logic** — `put_object()` call identical
- **File naming** — same `{date}-{princess}-{language}{-type}.mp3` pattern
- **ElevenLabs integration** — untouched
